# Sub-Session 06: PostgreSQL StatefulSet

This sub-session replaces the static PostgreSQL Deployment with a StatefulSet that creates its own PVC.

## Why StatefulSet Is Used

PostgreSQL is stateful because it stores database files. A normal Deployment can restart Pods, but it does not give each Pod a stable network identity or a predictable storage claim.

A StatefulSet is designed for workloads that need:

- Stable Pod names.
- Stable network identity.
- Stable storage per replica.
- Ordered creation and deletion.

For one PostgreSQL replica, the Pod name is stable:

```text
postgres-stateful-0
```

If the Pod is deleted, Kubernetes recreates a Pod with the same name and reattaches the same PVC.

## Why A Headless Service Is Used

The StatefulSet uses:

```yaml
serviceName: postgres-headless
```

The headless Service has:

```yaml
clusterIP: None
```

This gives StatefulSet Pods stable DNS records instead of load-balancing through one virtual IP.

For this lab, Flask still connects through the normal `postgres` ClusterIP Service. The headless Service exists for the StatefulSet identity.

## Why volumeClaimTemplates Is Used

`volumeClaimTemplates` lets a StatefulSet create one PVC for each Pod replica.

The generated PVC name will be:

```text
postgres-data-postgres-stateful-0
```

That PVC uses:

```text
storageClassName: ebs-gp3
```

This means the flow is:

```text
StatefulSet Pod -> volumeClaimTemplates -> PVC -> StorageClass -> EBS-backed PV
```

## Objects Created

The manifest creates:

- `postgres-headless`: headless Service for StatefulSet identity.
- `postgres`: normal ClusterIP Service used by Flask.
- `postgres-stateful`: StatefulSet that runs PostgreSQL.
- `postgres-data-postgres-stateful-0`: PVC generated by the StatefulSet.

## Before Applying

If the static PostgreSQL Deployment is running, remove it first:

```bash
kubectl delete -f subsessions/03-postgres-with-static-pvc/ --ignore-not-found
```

Apply the StorageClass sub-session first if the cluster does not already have `ebs-gp3`:

```bash
kubectl apply -f subsessions/05-storageclass/
```

## Apply

From `sessions/07-storage-pv-pvc-statefulset`:

```bash
kubectl apply -f subsessions/06-postgres-statefulset/
```

## Check

```bash
kubectl get statefulset -n app-storage
kubectl get pods -n app-storage -l app=postgres,storage-demo=stateful
kubectl get service -n app-storage
kubectl get pvc -n app-storage
kubectl get pv
```

Describe the StatefulSet:

```bash
kubectl describe statefulset postgres-stateful -n app-storage
```

Inspect the generated PVC:

```bash
kubectl describe pvc postgres-data-postgres-stateful-0 -n app-storage
```

## Persistence Test

Deploy Flask if it is not already running:

```bash
kubectl apply -f subsessions/04-flask-with-persistent-db/
```

Add messages through the UI. Then delete PostgreSQL:

```bash
kubectl delete pod postgres-stateful-0 -n app-storage
kubectl get pods -n app-storage -w
```

When PostgreSQL is ready again, refresh the app. The messages should still exist.

## Cleanup Note

StatefulSet PVCs are retained after the StatefulSet is deleted. Delete them explicitly when the lab is finished:

```bash
kubectl delete pvc -n app-storage -l app=postgres,storage-demo=stateful
```

This retention is intentional. Kubernetes avoids deleting state automatically because the PVC may contain production data.
